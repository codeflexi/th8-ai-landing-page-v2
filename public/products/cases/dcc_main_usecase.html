TH8 Advanced Use Cases

5 รูปแบบความเสี่ยงในธุรกิจไทย และ Logic การตรวจจับ

Source-to-Pay

1. Maverick Buying & Price Creep

การแอบซื้อของแพงกว่าสัญญา หรือ Supplier แอบขึ้นราคาเนียนๆ (1-2%)

Rule-Based Vector AI
Logic:
1. Vector Search: Match "A4 Paper Double A" (Invoice) <-> "Office Paper A4" (Contract Master)
2. Compare: IF Invoice_Price > Contract_Price THEN Alert
3. Trend: IF Price increase > 3% in 3 months (Price Creep) THEN Alert
Thai Context: โรงงานต่างจังหวัดชอบซื้อร้านข้างนอก (ร้านเพื่อน) โดยอ้างว่า "ของขาด" หรือ "ชื่อไม่เหมือนในระบบ" AI จะช่วยจับคู่ชื่อสินค้าที่เขียนไม่เหมือนกันได้
Logistics & Supply Chain

2. Phantom Accessorials

ค่าใช้จ่ายแฝงที่ตรวจสอบยาก (ค่ายก, ค่ารอ, ค่าล้างตู้)

LLM Analysis Rule-Based
Logic:
1. LLM Scan: อ่าน Description หายอดที่เป็น "Service" หรือ "Surcharge" (เช่น 'ค่าผ่านทาง', 'ค่าล่วงเวลา')
2. Validation: IF Surcharge > 0 AND No_Attachment (GPS/Receipt) THEN Block
3. Ratio Check: IF Surcharge > 15% of Base_Freight THEN Flag Manager
Thai Context: ใบวางบิลขนส่งไทยมักมีรายการ "Kha Borikan" (ค่าบริการ) ที่คลุมเครือ TH8 จะบังคับให้แจกแจงและขอหลักฐาน
Tax & Compliance

3. WHT Error (Withholding Tax)

หัก ณ ที่จ่ายผิดประเภท หรือลืมหัก (โดนค่าปรับย้อนหลัง)

LLM Reasoning
Logic:
1. LLM Interpret: "ค่าจ้างทำของ" (3%) vs "ค่าเช่า" (5%) vs "ค่าโฆษณา" (2%) vs "ค่าขนส่ง" (1%)
2. Cross Check: เปรียบเทียบกับยอด WHT ใน Invoice ที่ Vendor คำนวณมา
3. Alert: IF Calculated_Rate != Vendor_Rate THEN "Potential Tax Penalty"
Thai Context: เป็น Pain Point ใหญ่สุดของ AP ไทย เช่น "ค่าเช่าเครื่องถ่ายเอกสารพร้อมคนดูแล" ต้องแยกค่าเช่า (5%) กับค่าบริการ (3%) หรือเหมา? AI ช่วยตัดสินได้แม่นยำขึ้น
Fraud Prevention

4. Fuzzy Duplicate Invoice

การเวียนเทียนใบแจ้งหนี้ หรือเปลี่ยนเลขที่บิลนิดเดียว

Rule-Based Fuzzy Logic
Logic:
1. Exact Match: Vendor + Amount + Date (Basic)
2. Fuzzy Match: Vendor + Amount (Same) + Inv_No (Similarity > 90%)
Ex: INV-2024-01 vs INV/2024/01 vs 202401 (AI มองว่าเป็นใบเดียวกัน)
Thai Context: ปัญหา Coding ผิด หรือ Vendor ส่งมาทั้ง PDF และ Paper แล้วบัญชีเผลอ Key เข้าไปทั้งคู่
Inventory Control

5. Short Shipment (GR Mismatch)

จ่ายเงินเต็มจำนวน แต่ของเข้าคลังไม่ครบ

Rule-Based
Logic:
1. Fetch GR: ดึงข้อมูล Goods Receipt (GR) จาก Warehouse
2. Compare: IF Invoice_Qty > GR_Accepted_Qty THEN Block
3. Tolerance: ยอมรับ Variance ได้ไม่เกิน 0.5% (สำหรับสินค้าชั่งน้ำหนัก เช่น แป้ง/น้ำตาล)
Thai Context: ในธุรกิจอาหาร สินค้าพวก Bulk (น้ำตาล/น้ำมัน) มักมีน้ำหนักหายระหว่างทาง (Shrinkage) ต้องคุมเข้มไม่ให้จ่ายเงินส่วนที่หายไป
